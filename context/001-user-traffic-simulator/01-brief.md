# Brief: user-traffic-simulator

> Source: requirements-file (context/001-user-traffic-simulator/00-requirements.md) + P0 결정(Q-001=B, Q-002=A, Q-003=A+B)

가상 유저 다수가 "정해진 행동 그래프" 안에서 실제 사람처럼 SUT(System Under Test)를 돌아다니고,
설정한 확률로 이탈하며, 특정 API에 트래픽을 몰아 — 실 유저 없이도 개발자·기획자·디자이너가
시나리오/이탈/부하/놓친-에러 4관점의 이슈를 셀프서비스로 발견하는 웹 기반 실험 도구.

---

## 목표

1. **실 유저 없이 이슈 조기 발견** — 가상 유저로 SUT의 행동·부하·이탈 상황을 배포 전에 재현해, 동시성·가용성·스케일 한계·개발자가 놓친 에러를 드러낸다. (측정: 실험 런 1건당 분류된 이슈 신호 ≥1종 산출)
2. **비개발자 셀프서비스** — 개발자뿐 아니라 기획자·디자이너도 코드 없이 웹 UI만으로 실험을 구성·실행·해석한다. (측정: 비개발자가 실험 1건을 생성→실행→리포트 열람까지 외부 도움 없이 완료)
3. **4관점 비교 리포트** — 시나리오 준수 / 랜덤 이탈 / 부하 집중 / 놓친 에러를 하나의 리포트에서 비교 가능하게 제공한다. (측정: 단일 리포트에 3개 실행 모드의 지표가 동일 축으로 병치)

## 성공 지표

- **수집 완전성** — 실험 런 1건이 클라이언트측 지표(요청 수, 성공/실패율, status 분포, p50/p95/p99 지연, timeout 수, 에러율)를 누락 없이 산출.
- **부하 정확도** — 부하 집중 모드에서 목표 API의 실제 RPS/동시성이 설정한 가중·램프 프로파일을 합리적 오차(예: ±10%) 내로 추종.
- **이탈 충실도** — 설정한 이탈 확률(%)이 실제 실행 분포와 통계적으로 일치하며, **의존 엣지 위반 0건**(프로세스 건너뛰기 금지 제약 준수).
- **놓친-에러 표면화** — payload 변형/경계값 주입이 정상 순서로는 발생하지 않던 4xx/5xx/계약 위반을 ≥1건 드러냄 (goal #4 검증).
- **비개발자 완주(정성)** — 비개발자 1명이 웹 UI만으로 실험 생성→실행→리포트 해석을 외부 도움 없이 완료.

## 유저 플로우

**운영자(개발자/기획자) — Happy path**
1. 대상 환경 등록 — dev/staging 호스트 화이트리스트에 SUT 등록
2. API + payload 템플릿 등록 — 호출 대상 엔드포인트와 요청 본문 틀 입력
3. 행동 그래프 작성 — 상태 + 전이확률 + 의존 엣지(필수 선행). *또는* 경량 모드(가중 랜덤워크)로 빠르게 시작
4. 실험 파라미터 설정 — 가상 유저 수 / 이탈 확률(%) / 부하 집중 대상 API + 램프·스파이크 프로파일
5. (opt) 서버측 메트릭 소스 연결 — Prometheus/OTel/APM/로그 (근본원인 관측이 필요할 때)
6. 실험 실행 — 진행 상태 관찰
7. 리포트 열람 — 3모드(준수/이탈/부하) 지표 비교 + 분류된 이슈 목록

**뷰어(디자이너/기획자) — 분기**
- 공유된 리포트 URL 열람 → 이슈 목록·지표·타임라인을 읽기 권한으로 확인 (실험 설정 권한 불요)

**핵심 분기**
- *이탈*: 전이 시점에 확률적으로 선택-단계 생략 / 독립-단계 재정렬 / payload 변형 — 단 의존 엣지가 만족될 때만(선행 위반 금지)
- *부하 집중*: 특정 API로 가상 유저를 가중 funnel + 램프/스파이크 시계열 프로파일 적용
- [NEEDS CLARIFICATION: 실험 중단(kill switch) 트리거 — 수동 정지만인가, 자동 임계치(예: 에러율 > X%, 대상 5xx 급증)도 두는가? Q-009 안전장치와 연결]

## 리스크

- **[CRITICAL] prod 오발사 = self-DoS** — 트래픽을 *의도적으로* 몰리게 하는 도구라 대상 오지정 시 그대로 서비스 장애. 본 도구의 blast radius가 곧 핵심 기능. **완화**: dev/staging 기본 + 호스트 화이트리스트 + 하드 rate cap + kill switch (요구사항 Q-009=A 권장). 회사 정책 §1(prod 변경 절차)·§5.2(prod 테스트 금지) 직접 연결.
- **규모 천장** — 단일 노드 async(Q-008=B 방향)가 목표 피크를 못 내면 goal #3(서버 포화)을 유발조차 못 함. *가정*: 수백~수천 동시 유저. [NEEDS CLARIFICATION: 목표 피크 동시 유저/RPS 수치 — 확정 시 단일 노드 충분성 판정, Q-008 잠금 해제]
- **그래프 작성 진입장벽** — Q-002=A는 사용자가 상태 그래프를 직접 작성해야 해 비개발자(목표 2)에게 부담. **완화**: 경량 랜덤워크 모드(D) + 템플릿/예시 갤러리 + 그래프 비주얼 에디터.
- **관측 깊이 한계** — 클라이언트측만으론 "서버다운/스케일아웃"의 근본원인은 *추론*에 그침. **완화**: 서버측 opt-in(Q-003=B). *가정*: v1은 증상 표면화로 가치 성립, 근본원인은 후속 레이어.
- **PII 노출** — payload 템플릿에 실제 사용자 데이터가 들어가면 로그·리포트에 노출 위험(정책 §5.3). **완화**: 합성/마스킹 데이터 권장. [NEEDS CLARIFICATION: payload에 실데이터 허용 여부 + 마스킹 정책 — 정책 §5.3의 마스킹 도구 미정(TODO)과 연결]
- **결정성 vs 현실성 갭** — 그래프 모델은 실제 사람만큼 다양하지 않아 "실제 유저처럼"의 기대와 차이. *가정*: 인프라/로직 버그 탐색에는 충분(대화적 현실성은 불필요). 후속 LLM 고충실도 모드(Q-002 C)로 보완 가능.

## 의존성

- **대상 SUT(dev/staging)** — 실험을 쏠 실행 중인 시스템. *외부 승인*: 본인 통제 환경이면 없음. 단 **타 팀 소유 SUT면 부하 테스트 사전 합의 필요**.
- **(opt) 서버측 메트릭 소스** — Prometheus / OpenTelemetry / APM / 로그 엔드포인트 (Q-003 B 레이어 사용 시에만).
- **기술 스택 결정** — 웹 UI + 코어 그래프 실행 엔진 + 부하 워커의 언어/프레임워크/배포 형태. 현재 미정 → **tech-spec에서 확정** (P1 Q-004~Q-008 흡수).
- **회사 정책 준수** — §1(prod 변경 절차), §5.2(prod 테스트 금지), §5.3(PII 마스킹) — 안전장치·데이터 처리 설계에 반영 필수.
- 검토 결과: **외부 팀 API 의존·승인 절차 없음**(그린필드 사내 도구). 본 도구가 의존하는 유일한 외부 입력은 "사용자가 지정한 대상 SUT" 자체.
